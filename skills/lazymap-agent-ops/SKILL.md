# LazyMap Agent Ops

## 适用场景

当任务涉及以下任一项时使用：
- LazyMap 阅读地图 Agent 发布到 GitHub 或 Zeabur
- 搜书质量、封面命中、作者命中治理
- 阅读地图内容深度提升
- 积分制产品文案与动作链路调整
- 需要把一次性修改沉淀为可复用 SOP

## 核心原则

1. 搜书和生成分开治理。
2. 用户前台不暴露内部成本和模型实现细节。
3. 中文界面优先，默认不做多语言分支。
4. 搜书优先返回“明确书目 + 作者”，不要返回脏候选。
5. 阅读地图不是摘要页，必须有结构、方法、时间线、关键句和阅读路线。

## 默认工作流

1. 先确认入口问题属于哪一类：
- 搜书
- 上传解析
- 地图生成
- 页面呈现
- 账户与积分
- 发布部署

2. 若是搜书问题，按顺序检查：
- 本地图库命中
- Google Books
- Open Library
- 封面是否命中真实书封
- 作者是否稳定

3. 若是地图内容问题，按顺序检查：
- grounding 是否拿到章节与金句线索
- 模块标题是否有判断性
- 方法卡数量是否足够
- quotes 是否像真正可摘记的句子

4. 若是发布问题，按顺序执行：
- `npm run build`
- `node --check server.js`
- 检查 `.env.local` 未入库
- git 提交
- 创建/推送远端
- Zeabur 环境变量同步

## 输出标准

### 搜书
- 至少优先返回可识别作者的候选
- 错封面宁可用标题封面兜底，也不要给错图

### 地图
- 必须有更深的编辑感
- 不接受普通摘要式标题
- 方法卡要可迁移
- 金句要短、稳、可引用

### 产品文案
- 用户只看动作，不看内部机制
- 用户只看“上传生成 50 / 全网搜索生成 150”

## 常用检查

```bash
npm run build
node --check server.js
curl -s http://127.0.0.1:8787/api/health
```
